# 자동 배관 라우팅(Auto-Routing) C# 라이브러리 개발 계획서 (수정안)

본 계획서는 `GroupPatternViewer`를 라우팅 클라이언트로 삼아, 데이터베이스 환경 기반에서 장애물 복셀(Voxel) 맵을 구축하고 기존 설계(다발/패턴)를 추종하는 자동 배관 라우팅(A*) 라이브러리의 설계 및 구현 방안을 정의합니다.

## 목표 및 주요 변경 요구사항
1. **라우팅 클라이언트 연동**: `GroupPatternViewer`가 클라이언트가 되어 DB 데이터를 불러와 라우팅 엔진에 주입하고, 결과를 3D로 시각화.
2. **Voxel 점유 맵 생성**: DB에서 읽은 장애물(기둥, 박스) 및 타 배관 데이터를 기반으로 3D 공간을 복셀(그리드) 단위로 쪼개어 충돌/점유 맵을 자동 생성.
3. **기존 설계 추종형 A* 알고리즘**: 단순히 텅 빈 공간의 최단 거리를 찾는 것이 아니라, 기 추출된 **기존 설계 데이터(그룹 패턴 및 다발 세그먼트)의 궤적을 쫓아가도록 가중치를 부여**하는 알고리즘 구현.
4. **직교형 A* 적용 + PoC 구간 예외**: 장비 연결부(PoC)와 종단 덕트 연결부의 짧은 진입/진출 구간만 사선(Diagonal)을 허용하고 나머지는 완벽한 3D 직교(Orthogonal) 라우팅 수행.

## 1. 라이브러리 및 모듈 구조

`GroupPatternViewer` 솔루션 내에 라이브러리(가칭 `AutoRoutingLibrary`)를 구성하거나 하위 폴더 형태로 포함시킵니다.

### 주요 클래스 및 역할
* **`ObstacleSpatialIndex` (Voxel Map 대체 / 최적화 모듈)**
  * [메모리/속도 최적화]: 50mm 해상도의 거대한 공간을 단순 3차원 배열(Dense Array)로 만들면 수억 개의 Cell이 생성되어 메모리 오버플로우 및 속도 저하가 발생합니다.
  * 이를 해결하기 위해 공간을 **희소 해시맵(Sparse Spatial Hashing)** 으로 관리하거나, 아예 Voxel을 생성하지 않고 장애물들을 **R-Tree 또는 Octree 기반의 BoundingBox(AABB) 트리**로 구성합니다. A* 탐색 중 필요한 좌표만 동적으로 충돌 여부를 O(log N) 속도로 빠르게 검사합니다.
  * **[중요 예외 규칙]**: 장애물 데이터 중 **그레이팅(Grating), 바닥(Floor), 격자보** 형태의 객체는 시각적으로만 표시될 뿐 배관이 관통할 수 있으므로 트리 등록 및 충돌 검사에서 **제외(Pass-through)** 시킵니다.
* **`PatternGuidanceField` (기존 설계 추종 모듈)**
  * DB에서 읽은 `TB_GROUP_SEGMENTS` 데이터를 활용하여, 기준 다발이 지나가는 주변 공간 복셀들에 '보너스 가중치(선호도)'를 부여합니다. (경로 탐색 시 이 영역을 우선적으로 지나가려 함)
* **`AStarRouter` (핵심 알고리즘 엔진)**
  * 출발점(Start PoC)과 도착점(End PoC) 좌표를 받습니다.
  * `VoxelMap`과 `PatternGuidanceField`를 참조하여 A* 경로 탐색을 수행합니다.

## 2. 기존 설계 추종 알고리즘 상세 설계

기존 A* 알고리즘의 노드 간 이동 비용(Cost) 함수를 다음과 같이 커스텀합니다.

**Cost 함수** = `기본 이동 거리(G)` + `맨해튼 목적지 거리(H)` + `엘보우 패널티` - **`패턴 추종 보너스(Guidance)`**

### A. 패턴 추종 (Guidance)
* 기존 설계된 다발(수직/수평 세그먼트)의 중심축(`REP_X, Y, Z`) 주변의 일정 반경(예: 500mm 이내) 공간을 '설계 고속도로'로 지정합니다.
* 이 공간에 속한 복셀 노드는 이동 시 G 비용을 깎아주어(보너스 부여), 알고리즘이 스스로 다른 빈 공간으로 가지 않고 **기존 배관 뭉치 곁으로 합류하여 나란히 달리도록 유도**합니다.

### B. 직교 이동 및 메모리/속도 최적화 (Sparse A*)
* 상/하/좌/우/전/후의 **6방향 직교(Orthogonal)**로만 확장을 진행합니다.
* 엘보우(방향 전환)가 발생하면 꺾임 패널티 비용을 추가하여 잦은 꺾임을 방지합니다.
* **대공간(Empty Space) 탐색 최적화**: 텅 빈 공간에서 50mm 단위로 찔끔찔끔 전진하면 속도가 너무 느려집니다. JPS(Jump Point Search) 개념을 차용하거나, 빈 공간에서는 500mm 단위의 **Macro-step(성큼 걷기)**으로 빠르게 전진하다가, 장애물이나 목표물이 가까워지면 50mm 단위의 **Micro-step**으로 해상도를 높이는 **다해상도(Hierarchical) 탐색 기법**을 적용하여 연산량을 대폭 삭감합니다.
* Octree/R-Tree에서 간섭이 감지된 좌표나, 패턴 추종 중 간섭이 발생하는 노드는 탐색에서 원천 차단합니다.

### C. PoC 진입/진출 (사선 허용)
* 장비 노즐(Start PoC)의 임의 좌표에서 가장 가까운 비점유(Empty) 정규 복셀까지 다이렉트(사선) 선분 연결.
* 도착지 덕트 측(End PoC) 또한 인접한 최종 복셀에서 덕트 타겟 좌표까지 다이렉트(사선) 연결.

## 3. 개발 마일스톤 (Milestones)

1. **Voxel 점유 맵 기초 구축**: `GroupPatternViewer` 내 장애물/기존배관 데이터를 활용한 3D 그리드(Voxel) 매핑 및 시각적 검증 기능 개발.
2. **기존 설계 추종 필드 생성**: 다발 세그먼트 주변 공간에 A* 알고리즘용 보너스 점수(Cost Map) 필드 구축.
3. **직교형 A* 엔진 & PoC 사선 연결 구현**: A* 코어 엔진과 패널티, 사선 연결 예외 처리 구현.
4. **UI 연동 및 라우팅 테스트**: 뷰어 내에서 **"임의 장비 PoC ➔ 덕트 타겟 PoC 자동 라우팅 실행"** 버튼을 만들어, 3D 경로 튜브를 실시간 렌더링.
